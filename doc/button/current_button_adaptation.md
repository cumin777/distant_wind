# 按钮适配记录

## 当前状态

主菜单目前只有第一个按钮使用了自定义按钮：

- `START`：自定义 `GameButton`
- `Continue`：Godot 原生 `Button`
- `Quit`：Godot 原生 `Button`

自定义按钮场景：

```text
scenes/ui/GameButton.tscn
```

相关脚本：

```text
scripts/ui/BaseButton.gd
scripts/ui/GameButton.gd
```

## 当前为什么这样实现

`BaseButton.gd` 目前继承的是 Godot 原生 `Button`。

这是当前学习阶段的折中做法。原生 `Button` 继续负责比较可靠的交互层：

- 鼠标点击检测
- `pressed` 信号
- 禁用状态
- 基础键盘/手柄焦点能力

我们的自定义场景和脚本先主要负责视觉表现：

- 自定义子节点结构
- 鼠标悬停时放大
- 按下时缩小
- 发光显示
- 颜色和样式变化
- 自定义文字显示

也就是说，现在的结构是：

```text
Button 根节点
  -> 使用 Godot 原生交互
  -> 使用自定义子节点做视觉
  -> 使用脚本做动画反馈
```

## 为什么暂时没有继续使用 Control 根节点

长期来看，更自由的按钮设计应该是：

```text
Control 根节点
  -> 自己处理输入
  -> 自己发出 pressed / released 信号
  -> 自己处理焦点
  -> 获得完整视觉自由度
```

这个方向更接近参考项目的做法。不过，如果用 `Control` 当按钮根节点，就必须自己重新实现一套足够可靠的按钮行为：

- 鼠标进入/离开
- 鼠标按下/释放
- 鼠标移出后是否取消点击
- 禁用状态
- 键盘确认
- 手柄焦点
- 与现有 UI 代码兼容的信号
- 子节点不要拦截鼠标输入
- 布局尺寸和点击区域必须正确

第一次实验直接切到 `Control` 根节点后，暴露了一个实际问题：按钮不能稳定触发原来的主菜单逻辑。

所以当前版本先缩小范围，保留原生 `Button` 作为交互层，只把自定义能力放在视觉表现层。

## 后续演进方向

后续按钮系统可以拆成两层：

```text
BaseClickable.gd
  纯 Control 实现的交互驱动层

GameButton.gd
  视觉样式和动画表现层
```

建议后续步骤：

1. 先只保留 `START` 作为自定义按钮实验对象。
2. 在使用原生 `Button` 根节点的前提下，先练习视觉表现。
3. 等视觉层比较熟悉后，再单独做一个 `Control` 根节点的原型。
4. 让 `Control` 原型发出和现有菜单脚本兼容的 `pressed` 信号。
5. 测试稳定后，再只替换 `START` 按钮。

## 核心设计原则

最终目标不是简单地“替换 Godot 原生按钮”。

真正的目标是把按钮拆成几个清晰的职责：

```text
交互驱动
视觉结构
动画反馈
游戏内具体行为
```

当前实现是一个安全的中间阶段。它先保证游戏能正常点击运行，同时为后续演进到更自由的自定义按钮系统留下空间。
